## Context

当前 Tomato Hunter 的任务完成流程：每个番茄结束后进入 ReviewPhase → Settlement（展示掉落物）→ RestScreen（休息）。当怪物 HP 归零时，`damageTask()` 自动将状态设为 `killed`。但这个自动判定存在问题：用户可能低估了任务复杂度（HP 归零但任务未完成），也可能高估了（只做了 2 个番茄就完成了 4 HP 的任务）。击杀权应该交给用户。

DailyPlanBoard 当前是一个纯待办列表：展示所有已规划的任务，每个任务有完成进度条（completed/planned），支持排序和移除。但缺少：已完成区域、整体进度概览、用时对比反馈。任务完成后仍留在列表原位，用户无法感知今天的推进节奏。

## Goals / Non-Goals

**Goals:**
- 让击杀成为用户主动判断的行为，而非 HP 归零的自动结果
- 当 HP 归零时向用户确认，支持追击（重设血量）继续战斗
- 让用户可以在 DailyPlanBoard 中随时手动击杀已完成的任务
- 让任务完成（怪物击杀）成为一个有仪式感的高光时刻
- 让用户在 DailyPlanBoard 中实时感知今日进度和节奏
- 展示用时对比（实际 vs 计划），帮助用户校准估时能力

**Non-Goals:**
- 不改变现有的掉落物生成逻辑（loot.ts）
- 不重新设计 Settlement 的基础布局
- 不改变 unidentified → ready 的前半段状态机

## Decisions

### 1. 击杀权归用户：damageTask 不再自动 kill

**选择**：修改 `damageTask()` —— 当 HP 降到 0 时，不再自动设置 `status = 'killed'`，而是保持 `hunting` 状态，HP 为 0。在结算阶段（Settlement）向用户展示确认 UI。

**替代方案**：保持自动 kill，但在 Settlement 后加一步"撤销 kill"。

**理由**：从源头改比事后撤销更干净。自动 kill 后再撤销需要恢复 HP、回退状态，逻辑更复杂。让 HP 归零成为一个"信号"而非"判决"，用户决定最终结果。

### 2. 两条击杀路径

**选择**：

- **路径 A — Settlement 确认击杀**：HP 归零后，Settlement 展示"怪物倒地"+ 确认按钮 "确认击杀" / "发起追击"
- **路径 B — DailyPlanBoard 手动击杀**：用户在待办面板对任何 hunting/ready 任务点击"击杀"按钮，无需 HP 归零

两条路径都触发相同的庆祝流程。

**理由**：路径 A 处理"做到计划量"的情况，路径 B 处理"提前完成"的情况。两者覆盖了用户实际使用的全部场景。

### 3. 追击机制：重设 HP 继续战斗

**选择**：当用户在 HP 归零确认时选择"发起追击"，消耗一个道具（追踪术卷轴），弹出 HP 重设界面。新 total_hp 不得低于 actual_pomodoros（已造成伤害），current_hp = new_total_hp - actual_pomodoros。然后进入休息或回村。

**替代方案**：不用道具，直接免费追击。

**理由**：追击本质上是"估时失败"的修正。用道具增加少许摩擦力，鼓励用户在创建任务时更认真估时，同时不阻止修正。追踪术卷轴作为基础道具，可通过掉落获取，不稀缺。

### 4. 击杀庆祝：在 Settlement 中增加条件分支

**选择**：击杀确认后，在 Settlement 组件中渲染增强版庆祝视图（统计卡片 + 视觉效果）。手动击杀走独立的弹窗庆祝。

**理由**：Settlement 承担"战斗结果展示"职责，击杀庆祝是其自然延伸。

### 2. DailyPlanBoard 三区设计：进行中 / 待战 / 已讨伐

**选择**：将 plan entries 按任务状态分为三个可视区域，由 planStore 的计算属性驱动。

**替代方案 A**：仅在已完成任务上叠加"已完成"徽章，不做分区。
**替代方案 B**：将已完成任务直接隐藏。

**理由**：
- 方案 A 无法提供"清单在缩短"的进度感
- 方案 B 会让用户失去"今天做了什么"的回顾能力
- 三区设计同时满足进度感（待战区缩短）和成就感（已讨伐区增长）

### 3. 进度计算：基于 planStore 现有数据

**选择**：从 `planned_task_entries` 聚合计算总进度，不引入新的数据模型。

- 总计划番茄 = Σ entry.planned_pomodoros_today
- 已完成番茄 = Σ entry.completed_pomodoros_today
- 任务分区依据 = entry 关联的 task.status（killed → 已讨伐，hunting → 进行中，ready → 待战）

**理由**：所有需要的数据已经存在，只需在 store 层添加 getter/computed 属性。

### 4. 用时对比：简洁的视觉标记

**选择**：在已讨伐区的每个任务上显示"实际/计划"番茄数，超时用红色、省时用绿色、刚好用默认色。

**理由**：用户需要的是快速的视觉反馈，不是详细的时间分析报告。红/绿标记在像素风格中也容易实现。

### 5. 击杀庆祝的额外奖励

**选择**：击杀时不额外增加掉落物数量，而是通过视觉升级（更大的动画、闪光效果、击杀统计摘要）来制造仪式感。

**理由**：掉落物经济已经在 economy-balance spec 中平衡过。仪式感更多来自视觉和反馈而非数值。

## Risks / Trade-offs

- **damageTask 行为变更影响面大** → 核心击杀逻辑改变。缓解：变更局限于 HP=0 时的行为分支，HP>0 的正常伤害路径不变。
- **追击道具可能断供** → 如果用户道具用完无法追击。缓解：追踪术卷轴设为常见掉落物或基础商店可购买。
- **DailyPlanBoard 手动击杀可能误操作** → 缓解：手动击杀需要二次确认对话框。
- **用时对比可能给用户带来压力** → 缓解：仅在已讨伐区展示，使用轻柔的颜色差异而非强烈的警告。超时文案用"多花了 N 个番茄 💪"鼓励语气。
