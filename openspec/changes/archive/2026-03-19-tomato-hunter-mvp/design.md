## Context

Tomato Hunter 是一个全新项目，从零开始构建。目标是将番茄工作法包装在怪物猎人式的游戏隐喻中，解决效率工具长期使用中体验单调、缺乏成长感的问题。

当前状态：项目目录已初始化，有 OpenSpec 脚手架但无代码。已完成完整的产品设计和实施计划。

约束：
- 单人 MVP，社区/多人功能后续阶段
- Windows 桌面端优先
- 数据全部本地存储（SQLite），无后端服务
- AI 功能可选，离线可用

## Goals / Non-Goals

**Goals:**
- 构建完整的"村庄→狩猎→结算→回村"核心循环
- 实现 GTD 式任务管理，包装在游戏语言中
- Rust 侧精准计时器，不受前端渲染影响
- 装备系统提供策略深度，驱动技能/策略成长而非数值膨胀
- 浮窗模式在专注期间最小化干扰

**Non-Goals:**
- 多人协作 / Boss 战（第二阶段）
- 社区功能、匹配组队（第二阶段）
- 成就勋章系统（第二阶段高优）
- macOS 构建（跑通 Windows 后再做）
- AI 图像生成猎物视觉（MVP 用 emoji/图标）
- 数据云同步、统计报表
- 等级系统、经验值、天赋树、数值膨胀

## Decisions

### D1: Tauri v2 + React + TypeScript

**选择**: Tauri v2 桌面壳 + React 前端 + Rust 后端
**替代方案**: Electron (更重，但生态更大), PWA (无系统集成能力), 纯原生 (开发成本过高)
**理由**: Tauri 轻量、安全、Rust 后端性能好。第一版只做壳 + Web 前端，不深度系统集成，验证核心玩法的速度和 PWA 差不多，但拿到了桌面端形态优势（独立窗口、托盘图标、不在浏览器标签页中）。

### D2: 计时器在 Rust 侧实现

**选择**: Rust 后端使用 `std::time::Instant`（单调时钟）计时，通过事件通知前端
**替代方案**: 前端 `setInterval` 计时
**理由**: 前端计时在窗口最小化或系统负载高时不可靠。Rust 单调时钟不受系统时间调整影响，精度可靠。

### D3: SQLite 本地存储（via tauri-plugin-sql）

**选择**: 前端通过 `@tauri-apps/plugin-sql` 直接执行 SQL，而非通过 Rust commands 中转
**替代方案**: 所有 CRUD 走 Rust Tauri commands
**理由**: MVP 阶段减少样板代码，前端直接操作 DB 开发效率更高。计时器等性能敏感操作仍走 Rust。

### D4: 浮窗 = Tauri 多窗口

**选择**: 狩猎时创建 always-on-top 小窗口，主窗口隐藏
**替代方案**: 单窗口内切换模式
**理由**: 浮窗独立于主界面，不占屏幕空间，用户可以同时看到自己的工作界面和狩猎进度。

### D5: AI 调用可选

**选择**: Claude API 生成怪物，失败或离线时回退到确定性名称生成器（Rust 侧）
**替代方案**: 仅使用 AI, 仅使用离线生成
**理由**: AI 增强体验但不是必须依赖。API key 未配置或请求失败时，自动切换到离线方案，不阻断用户流程。

### D6: 暂停需消耗道具

**选择**: 暂停消耗「烟雾弹」道具，单次上限 3 分钟，超时自动撤退，无道具不可暂停
**替代方案**: 自由暂停（无限制）, 不允许暂停
**理由**: 专注期间中断不被鼓励，但完全禁止暂停不够人性化。道具消耗在两者之间找到平衡——暂停有代价但可用。

## Risks / Trade-offs

**[Tauri v2 生态不够成熟]** → 可能遇到 plugin 兼容性问题。缓解：只使用官方 plugin（sql, notification），MVP 不做深度系统集成。

**[SQLite 前端直操作的安全性]** → 前端可以执行任意 SQL。缓解：纯本地应用，无网络攻击面，MVP 可接受。后续可迁移到 Rust commands。

**[AI API 费用和延迟]** → Claude API 每次生成怪物需要一次请求。缓解：使用 Haiku 模型（便宜快速），且有完整离线回退。

**[单文件 SQLite 的并发限制]** → 多窗口同时写入可能有锁冲突。缓解：写操作主要在主窗口，浮窗只读取计时器状态。

**[项目更名 Potato → Tomato]** → 现有设计文档中使用 "Potato Hunter"。缓解：代码中统一使用 "Tomato Hunter"，旧设计文档保留作参考。
