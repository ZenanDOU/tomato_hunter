## Context

Tomato Hunter 的核心循环（村庄→狩猎→结算→休息）已经完备，但玩家缺少长期进度感。已有数据（pomodoros、tasks、loot_drops、player_inventory）中蕴含成长轨迹，但从未被聚合展示。本设计在不引入新货币/数值系统的前提下，通过"猎人档案"面板将已有数据可视化为里程碑和收集进度。

现有村庄 Tab 结构：收件箱 | 狩猎 | 每日计划 | 工坊 | 农场。猎人档案将作为第六个 Tab 加入。

## Goals / Non-Goals

**Goals:**
- 让玩家感受到随时间推移的成长（里程碑 + 统计数字）
- 提供长期收集目标（怪物图鉴 15/15）
- 对已有玩家追溯计算历史成就
- 首次发现新物种时有即时反馈

**Non-Goals:**
- 不引入 XP/经验值系统
- 不引入新货币
- 不锁定任何现有功能（装备、计时器模式等）
- 不改变计时器或战斗核心逻辑
- 不做社交/排行榜功能
- 不做猎人等级（本次讨论中明确排除）

## Decisions

### D1: 里程碑用独立表存储，而非实时计算

**选择：** 新增 `milestones` 表，记录已达成里程碑的 ID + 达成时间 + 是否已展示通知。

**替代方案：** 每次打开档案页面实时查询计算。
**为何不选：** 需要判断"是否首次达成"来触发通知弹窗，纯实时计算无法区分"刚达成"和"之前就达成了"。存储层解决这个问题最干净。

### D2: 物种发现记录复用 tasks 表，不新建表

**选择：** 从 `tasks` 表聚合 `SELECT DISTINCT monster_variant FROM tasks WHERE status = 'killed'` 来获取已发现物种列表。首次发现时间 = 该 variant 最早的 completed_at。

**替代方案：** 新建 `species_discoveries` 表。
**为何不选：** tasks 表已包含所有必要信息（monster_variant 对应物种 ID、completed_at 对应发现时间），新建表是冗余的。聚合查询性能足够（最多几百条 task 记录）。

### D3: 里程碑检测在结算阶段触发

**选择：** 在每次 pomodoro 完成结算时（loot drop 之后、休息之前），运行里程碑检测。检测到新达成的里程碑写入 DB 并标记待展示，在结算画面的最后展示里程碑弹窗。

**替代方案 A：** 打开猎人档案时检测。
**为何不选：** 错过即时反馈的最佳时机——"刚完成第 10 次击杀"这个时刻应该被庆祝，而不是等玩家主动去看。

**替代方案 B：** 后台定时轮询。
**为何不选：** 过度复杂，番茄钟结算是唯一会改变里程碑相关数据的时间点。

### D4: 里程碑定义为前端常量，不存数据库

**选择：** 里程碑列表（ID、名称、描述、检测条件）定义为 TypeScript 常量数组。数据库只存"哪些已达成"。

**理由：** 里程碑是游戏设计内容，不是用户数据。放在代码中方便迭代和版本控制。新增里程碑只需加一行定义 + 上线后自动追溯检测。

### D5: 猎人档案作为村庄新 Tab，图鉴内嵌其中

**选择：** 猎人档案是一个 Tab 页面，包含三个区域：统计总览、里程碑列表、怪物图鉴。图鉴不是独立页面。

**理由：** 保持导航扁平。图鉴内容量不大（15 种怪物），不需要独立路由。

### D6: 追溯计算在 migration 中处理

**选择：** 新增 migration 创建 `milestones` 表后，通过 SQL 聚合查询已有数据，批量插入已达成的里程碑（marked as `notified = true` 跳过弹窗通知）。

**理由：** 老玩家打开更新后的 app 应该直接看到自己的历史成就，而不是被一连串弹窗轰炸。

## Risks / Trade-offs

**[里程碑检测性能]** → 每次结算需执行多条聚合查询（COUNT pomodoros, COUNT killed tasks, COUNT DISTINCT variants 等）。缓解：查询简单，数据量小（个人使用），且仅在结算时触发（每 25 分钟最多一次）。

**[里程碑定义扩展]** → 如果未来里程碑条件变得复杂（如"连续 7 天有击杀"），纯 SQL 聚合可能不够。缓解：检测函数设计为 `(db) => Promise<boolean>`，既可以用 SQL 也可以用 JS 逻辑。

**[monster_variant 作为物种 ID 的可靠性]** → 当前 monster_variant 是由 bestiary 系统确定性生成的物种标识符。如果 bestiary 逻辑变更导致 variant 值改变，图鉴历史数据会不一致。缓解：bestiary 物种 ID 已稳定（15 种固定），短期内不会变。

**[村庄 Tab 数量增加]** → 从 5 个增加到 6 个，移动端水平空间可能紧张。缓解：这是桌面 Tauri 应用，不存在移动端问题。
