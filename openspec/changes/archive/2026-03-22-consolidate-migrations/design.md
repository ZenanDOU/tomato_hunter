## Context

当前 `src-tauri/migrations/` 下有 13 个增量迁移文件，`db.rs` 逐一注册。项目处于开发阶段，无真实用户数据，增量迁移带来的兼容性保障没有实际价值，反而增加了理解成本。

最终态 schema 包含 13 张表、6 个索引、种子数据（11 材料 + 14 装备 + 默认设置）。

## Goals / Non-Goals

**Goals:**
- 将 13 个迁移合并为 1 个 `001_initial.sql`，直接描述最终态 schema
- 删除本地 SQLite 数据库，让应用从零重建
- 简化 `db.rs` 到只注册 1 个 migration

**Non-Goals:**
- 不改变任何表结构、约束、索引或种子数据的最终状态
- 不修改前端代码或 Rust command 层
- 不建立正式的生产迁移流程（那是发布前的事）

## Decisions

### 1. 合并方式：手动编写最终态 SQL，而非脚本拼接

**选择**: 手动编写一个干净的 `001_initial.sql`，按「建表 → 索引 → 种子数据」的逻辑顺序组织。

**理由**: 迁移之间有破坏性操作（005 DELETE + 重新 INSERT 装备、008 DROP TABLE 重建 tasks），机械拼接会引入冗余和冲突。手动编写可以产出最干净的结果。

**替代方案**: 从现有数据库 `.schema` 导出 → 但种子数据不在 schema 中，还是需要手动处理。

### 2. 删除本地数据库：手动删除 + 文档说明

**选择**: 在 tasks 中明确标注手动删除 `%APPDATA%/com.tomato-hunter.app/tomato_hunter.db*` 三个文件。

**理由**: 这是一次性操作，不需要自动化。Tauri 的 `tauri_plugin_sql` 在下次启动时会自动运行迁移重建数据库。

### 3. 追溯逻辑全部移除

**选择**: 009 的追溯 milestone 计算、010 的追溯装备解锁、005 的旧数据迁移逻辑——全部不纳入合并后的 SQL。

**理由**: 这些逻辑的唯一目的是处理「从旧 schema 升级」的场景。从空库建表不需要追溯。

## Risks / Trade-offs

- **[团队成员本地数据丢失]** → 仅个人开发项目，无影响。即使有协作者，开发数据也不关键。
- **[合并后 SQL 与当前运行态不一致]** → 合并后启动应用验证，确认所有表和种子数据正确创建。
